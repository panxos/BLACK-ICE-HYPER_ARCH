## 🔴 CRÍTICOS (7)

### C-01: Archivo LICENSE ausente
- **Archivos afectados:** Todo el repositorio
- **Descripción:** Tanto `README.md` como `README.en.md` declaran licencia MIT, pero **no existe ningún archivo LICENSE** en el repositorio. Esto deja el proyecto sin licencia legalmente válida.
- **Impacto:** Legal — cualquier uso del código carece de base legal clara.
- **Ubicación:** Raíz del proyecto

### C-02: Rutas hardcodeadas a `/home/faravena`
- **Archivos afectados:**
  - `tools/repair_wiki.sh` (líneas 4, 9, 41)
  - `tools/repair_wiki_simple.sh` (líneas 7, 15)
  - `tools/repair_wiki_robust.sh` (líneas 8, 16)
  - `dotfiles/bin/htb-connect.sh` (línea 11)
- **Descripción:** Rutas absolutas al usuario del author que romperán en cualquier otro sistema.
- **Ejemplo:**
  ```bash
  # tools/repair_wiki.sh:4
  LOG_FILE="/home/faravena/wiki_repair.log"

  # dotfiles/bin/htb-connect.sh:11
  VPN_CONFIG="${VPN_CONFIG_DIR}/htb.ovpn"
  ```
- **Impacto:** Los scripts fallarán en cualquier máquina que no sea la del autor.

### C-03: Bypass de verificación de firmas PGP en modo automático
- **Archivo:** `src/modules/02_base.sh`
- **Descripción:** El "Protocolo Nuclear" relaja el `SigLevel` de pacman a `Optional TrustAll` en el intento 3, y se habilita automáticamente en `AUTO_MODE`. Esto desactiva la verificación de firmas de paquetes.
- **Impacto:** Un ataque MITM podría inyectar paquetes maliciosos sin detección.

### C-04: Contraseña LUKS potencialmente vacía en modo automatizado
- **Archivo:** `src/modules/01_disk.sh`
- **Descripción:** Si `ENABLE_LUKS=yes` pero `LUKS_PASSWORD` no está definida, `LUKS_PASS` será vacío. `cryptsetup luksFormat` podría aceptar una passphrase vacía, creando un volumen "encriptado" sin protección real.
- **Impacto:** Falsa sensación de seguridad; datos potencialmente accesibles sin contraseña.

### C-05: Login SSH de root habilitado sin mecanismo de desactivación
- **Archivo:** `src/modules/03_config.sh`
- **Descripción:** Se configura `PermitRootLogin yes` en sshd_config. El comentario dice "temporalmente", pero no existe lógica posterior que lo revierta.
- **Impacto:** Root accesible por SSH post-instalación — superficie de ataque innecesaria.

### C-06: `post-install-validate.sh` verifica `yay` en lugar de `paru`
- **Archivo:** `tests/post-install-validate.sh` (línea ~37)
- **Descripción:** El script ejecuta `check_bin yay`, pero el proyecto migró de `yay` a `paru` (documentado en CHANGELOG v3.2.0). Esta prueba **siempre fallará** en una instalación limpia.
- **Impacto:** Test siempre da falso negativo; no valida el AUR helper real.

### C-07: Entrada duplicada en array de herramientas de seguridad
- **Archivo:** `src/deploy/02_security_tools.sh`
- **Descripción:** `responder` aparece **dos veces** en el array `ALL_TOOLS`:
  ```bash
  "responder|Responder - LLMNR Poisoner|AD"
  "responder|Responder - LLMNR Poisoner|AD"
  ```
- **Impacto:** Se instalará/listará dos veces; desperdicio de recursos y confusión.

## 🟠 ALTOS (6)

### H-01: 49 de ~57 scripts carecen de `set -euo pipefail`
- **Archivos afectados:** Casi todos los módulos en `src/modules/`, `src/deploy/`, y scripts en `dotfiles/bin/`
- **Descripción:** Solo 8 scripts tienen algún modo estricto. Los 49 restantes no tienen protección contra errores silenciosos, variables no definidas o fallos en pipes.
- **Scripts clave sin strict mode:**
  - `src/deploy/00_repositories.sh`
  - `src/deploy/01_hyprland_base.sh`
  - `src/deploy/02_security_tools.sh`
  - `src/modules/00_environment.sh`
  - `src/modules/01_disk.sh`
  - `src/modules/99_final.sh`
  - `tests/pre-install-check.sh`
  - `dotfiles/hypr/scripts/startup.sh`
  - `dotfiles/bin/theme_switcher.sh`
  - `dotfiles/bin/wallpaper_manager.sh`
- **Impacto:** Errores silenciosos pueden propagarse sin detección.

### H-02: Expansión de variables sin comillas
- **Archivos afectados:** Múltiples
- **Descripción:** Variables sin comillas que contienen paths o listas de paquetes fallarán con espacios:
  ```bash
  # src/modules/01_disk.sh
  pacstrap /mnt $PACKAGES          # $PACKAGES contiene espacios intencionalmente

  # src/modules/04_user.sh
  grub-install --target=i386-pc --recheck $DISK  # $DISK sin comillas

  # dotfiles/waybar/scripts/ethernet_status.sh
  ip -4 addr show $iface           # $iface sin comillas

  # src/deploy/99_finalization.sh
  sudo -u $CURRENT_USER xdg-user-dirs-update  # sin comillas
  ```
- **Impacto:** Fallo en paths con espacios; comportamiento indefinido.

### H-03: Contraseñas en variables de entorno (exposición por `/proc`)
- **Archivo:** `src/modules/04_user.sh`
- **Descripción:** Las contraseñas se pasan como `_BI_ROOT_PASS` y `_BI_USER_PASS` vía variables de entorno. Estas son visibles en `/proc/<pid>/environ` y pueden filtrarse en core dumps. Aunque se hace `unset` al final, la ventana de exposición existe durante `arch-chroot`.
- **Impacto:** Posible fuga de credenciales durante la instalación.

### H-04: NOPASSWD sudo para grupo wheel
- **Archivo:** `src/modules/04_user.sh`
- **Descripción:** Se configura `%wheel ALL=(ALL:ALL) NOPASSWD: ALL`, dando acceso root sin contraseña a todos los usuarios del grupo wheel.
- **Impacto:** Postura de seguridad significativa — debe estar documentada y ser configurable.

### H-05: `curl | bash` sobre HTTP (no HTTPS)
- **Archivo:** `bootstrap.sh` (línea 9)
- **Descripción:** Las instrucciones usan `curl -L http://is.gd/blackice | bash`. El shortener usa HTTP plano, susceptible a interceptación/modificación del redirect.
- **Impacto:** Un atacante en la red podría redirigir a un script malicioso.

### H-06: No hay verificación de existencia antes de `source`
- **Archivos:** `install.sh`, `deploy_hyprland.sh`
- **Descripción:** Ambos scripts hacen `source` de módulos sin verificar que existan primero. Si un archivo falta o es ilegible, el error será críptico.
  ```bash
  # install.sh — sin guardas
  source "$INSTALL_DIR/src/modules/00_environment.sh"
  source "$INSTALL_DIR/src/modules/01_disk.sh"
  # ...

  # deploy_hyprland.sh — igual
  source "$SCRIPT_DIR/src/deploy/00_repositories.sh"
  ```
- **Impacto:** Fallos crípticos si archivos se corrompen o faltan.

## 🟡 MEDIOS (8)

### M-01: Mismatch de versiones
- **Archivos afectados:**
  - `bootstrap.sh` — declara `Version: 1.0`
  - `src/lib/logging.sh` — banner hardcodea `v3.3.0`
  - Proyecto real está en **v3.3.1** (según CHANGELOG)
- **Impacto:** Confusión para usuarios y contribuidores sobre la versión real.

### M-02: Race conditions con delays fijos
- **Archivo:** `dotfiles/hypr/scripts/startup.sh`
- **Descripción:** Usa `sleep 3` después de lanzar `swaync &` y otros servicios. Delays fijos fallan en sistemas lentos o son innecesariamente largos en sistemas rápidos.
- **Impacto:** Servicios pueden no estar listos; o se desperdicia tiempo de inicio.

### M-03: `pkill -9` (SIGKILL) sin intento previo de SIGTERM
- **Archivo:** `dotfiles/bin/waybar_start.sh`
- **Descripción:** Usa `pkill -9 waybar` directamente, matando el proceso sin permitir cleanup graceful.
- **Impacto:** Posible corrupción de estado; pérdida de datos no guardados.

### M-04: `theme_switcher.sh` sin validación de entrada
- **Archivo:** `dotfiles/bin/theme_switcher.sh`
- **Descripción:**
  - Sin `set -euo pipefail`
  - `THEME=$1` sin validación — si se llama sin argumentos, cae en `*)` con `exit 1` pero sin `set -e` depende del caller
  - No verifica si `$THEMES_DIR` existe antes de usarlo
  - `pkill waybar` sin verificar si waybar está corriendo
- **Impacto:** Fallos silenciosos o errores confusos.

### M-05: Referencia inconsistente a directorio de wallpapers
- **Archivo:** `dotfiles/hypr/scripts/wallpaper_manager.sh`
- **Descripción:** Los comentarios y variable `WALLPAPER_DIR` apuntan a `$HOME/.config/hypr/wallpapers`, pero los scripts de deploy instalan en `$HOME/Pictures/wallpapers`.
- **Impacto:** El script podría no encontrar los wallpapers esperados.

### M-06: Directorio `assets/diagrams/` vacío
- **Ubicación:** `assets/diagrams/`
- **Descripción:** El directorio existe pero no contiene archivos. Debería poblarse o eliminarse.
- **Impacto:** Repo sucio; confusión sobre contenido esperado.

### M-07: `bootstrap.sh` pierde error trap al usar `exec`
- **Archivo:** `bootstrap.sh` (línea ~223)
- **Descripción:** Usa `exec ./install.sh` que reemplaza el proceso del shell. El ERR trap definido en `bootstrap.sh` se pierde, y `install.sh` debe tener su propio manejo de errores autónomo.
- **Impacto:** Si `install.sh` falla sin trap propio, el error no se captura limpiamente.

### M-08: `grep -oP` (regex Perl) no portable
- **Archivos:** Varios scripts (`ethernet_status.sh`, `utils.sh`, etc.)
- **Descripción:** `grep -oP` requiere GNU grep. No funcionará en BSD/macOS. Aceptable para proyecto Arch-only, pero limita portabilidad.
- **Impacto:** Baja portabilidad fuera de Linux con GNU grep.

- **Descripción:** Comentarios y mensajes de log en español, nombres de variables en inglés. Ejemplo:

### L-02: `ls | wc -l` en lugar de `find | wc -l`
- **Archivos:** Varios
- **Descripción:** Patrones como `ls -1 "$DIR" | wc -l` fallan con filenames que contienen newlines. Preferible: `find "$DIR" -maxdepth 1 -type f | wc -l`.
- **Impacto:** Extremadamente bajo — filenames con newlines son raros.

### L-03: Banner del README references imágenes que pueden no existir localmente
- **Archivo:** `README.md`
- **Descripción:** Referencias a imágenes del wiki de GitHub que pueden no estar presentes en clones locales.
- **Impacto:** README roto en clones locales sin acceso a las imágenes remotas.

### L-04: Inconsistencia en quoting style
- **Descripción:** Algunos scripts usan `readonly` y comillas dobles consistentemente (`bootstrap.sh`), otros no usan ni una ni otra.
- **Impacto:** Estética y mantenibilidad.

### L-05: Falta módulo `08_ai_tools.sh` en algunos flujos

   ```bash
       fi
   ```
